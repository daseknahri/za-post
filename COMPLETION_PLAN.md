# za-post — Completion Plan

> Generated by a 118-agent gap analysis (107 gaps found, 93 confirmed, 14 rejected as false alarms). This is the authoritative roadmap to a complete, shippable app.

## 0. How complete is it today?

The app is **functionally end-to-end but only verified at minimal scale** (one account, two groups, one post). The core engine works: the orchestrator runs cycle-based, account-centric batches; the worker launches per-account hidden Chromium profiles, logs in (cookies/credentials), opens the composer, types human-like captions with spintax variation, uploads hash-perturbed images, publishes, and adds a first comment. Persistence is durable (atomic temp→fsync→rename with `.bak` recovery and a serialized `update()` mutex), anti-spam pacing exists (daily caps, exponential rate-limit cool-down, comment delays, per-account proxies), and there's a working remote dashboard + optional Cloudflare tunnel and an opt-in VPS licensing system. The desktop UI covers all CRUD and run control.

The headline gaps are concentrated in five areas: **(1) Facebook fragility** — rate-limit/checkpoint/composer/post-button/comment detection all rely on hardcoded English/Hungarian text with no fallbacks or drift detection, so a single FB UI/locale change silently breaks posting; **(2) silent data loss** — image-upload failure publishes posts without their image, and `pcu-state.json` write failure can re-post published content after a crash; **(3) licensing is bypassable** — limits are UI-only against a permissive 9999-stub, offline fallback can bypass revocation, and the VPS server has no HTTPS/rate-limiting/health-check and stores keys plaintext; **(4) credentials/secrets at rest are plaintext** (FB passwords in `data.json`, tunnel token in logs); **(5) no automated test or CI/CD infrastructure** — only manual diagnostic scripts. None of these block a careful single-operator pilot, but all block a confident, sellable, at-scale shipment.

## 1. Definition of "complete"

The app is **done/shippable** when all of the following are true:

- [ ] **No silent data loss.** A post never publishes without its intended image; image upload retries then either succeeds or the group is skipped and reported as `error`/`incomplete`. Rotation/dealt-state, account-status, daily-count, and audit-trail writes either succeed or surface a visible failure (and rotation failure halts the run).
- [ ] **FB detection is resilient.** Composer, post-button, comment-box, rate-limit, checkpoint, and pending-approval detection each have multiple fallbacks (role/aria/text across locales) and emit a high-priority **selector-drift** alert when a previously-working selector returns zero matches, instead of silently skipping.
- [ ] **Licensing matches a decided model.** Either (a) limits are enforced in the backend (every mutation handler) against real license state and offline fallback fails-closed for customer keys, OR (b) the stub is formally documented as permanent and UI-only checks are removed. The VPS server has HTTPS, rate limiting, a `/health` endpoint, and a documented backup/rotation story.
- [ ] **Secrets are protected at rest and in transit.** FB credentials encrypted (DPAPI) or not persisted; tunnel token never logged in full; license traffic over HTTPS; `keys.json` encrypted or on an encrypted volume.
- [ ] **Inputs are validated.** Image URLs (drag/drop/paste + `downloadImage`) reject private IPs / non-image content-type / oversized payloads; proxy strings validate scheme + port range; cookie imports validate critical cookies.
- [ ] **Half-finished features are reachable or removed.** `finishAutomation`, `selectImage`, and the two backward-compat settings (`warmupRuns`, `rateLimitCooldownHours`) are either wired into the UI or deleted; `account-attention` updates the renderer in real time.
- [ ] **Quality gates exist.** A test runner (`npm test`) covers spintax, store concurrency, settings clamping, license logic, posting-mode rotation, cool-down, and FB-detection (HTML snapshots). A CI pipeline builds the portable zip and runs tests. The portable build is reproducible (no hardcoded winCodeSign version) and ideally code-signed.
- [ ] **Docs are accurate.** `OPERATIONS.md`, `README.md`, `HANDOFF.md`, and `READ-ME-FIRST.txt` reflect actual behavior (licensing opt-in, hardened migration defaults, multi-instance `--profile` isolation, env-var reference).
- [ ] **Verified at realistic scale** — a multi-account / multi-group / multi-cycle live run completes with an accurate audit trail.

## 3. Dependency / sequencing notes

- **M1-05 (license decision) gates M1-06, M3-04, M3-05, and part of M4-07/M4-08.** The owner's enforcement decision (§4) determines whether backend limit checks, fail-closed offline behavior, and VPS hardening are "required" or "optional/documented-away." Resolve this decision before touching any licensing code.
- **M2-01 (surface write failures) underpins M2-02 and M1-02.** Once `store.update`/`appendReport`/`setAccountStatus` stop swallowing errors, the rotation-halt (M1-02) and status-clobber (M2-02) fixes have a visible-failure substrate to build on. Do M2-01 early in M2.
- **M1-03 + M1-04 share a drift-alert mechanism.** Build the `selector-drift-detected` event/logging once (in M1-03) and reuse it for the detection hardening in M1-04. M4-02 (snapshot tests) depends on the refactored, fallback-bearing detectors from M1-03/M1-04 — do the code first, then the tests.
- **M4-01 (test runner) must land before M4-02, M4-06, and M4-03.** CI (M4-03) is only worth wiring once there's a `npm test` target to run; snapshot tests (M4-02) and IPC tests (M4-06) need the framework.
- **M4-04 (reproducible build) should precede M4-03/M4-05.** CI can't reliably produce the portable zip until the winCodeSign version is no longer hardcoded; signing (M4-05) plugs into the same build path.
- **M2-07 (settings clamp) pairs naturally with M4-01** (its tests) but the clamping code is independent and can ship first.
- **M3-01 (credential encryption)** requires an on-load migration of existing plaintext — sequence it so the migration runs before any code that reads credentials assumes ciphertext.

## 4. Risks & decisions needed from the owner

1. **License enforcement model (blocks M1-05/M1-06/M3-04/M3-05).** Is the VPS gate the real enforcement, and should `maxGroups`/`maxAccounts` become backend-enforced against real license state — or is the permissive stub permanent? What model: machine-bound seat, subscription/expiry, or per-account cap? This single decision changes ~4 items from "build" to "delete + document."
2. **Offline revocation tolerance.** Acceptable for the owner key to keep working offline indefinitely, and for customer keys to **fail-closed** on a slow/unreachable server? (Recommended: fail-closed for customer keys.) Also: what is the plaintext owner key behind the `OFFLINE_ALLOW` hash, is `144.91.127.7:3509` still live, and is there an HTTPS endpoint to switch `DEFAULT_SERVER` to?
3. **Anti-detection strategy.** What Facebook flag rate is acceptable, and should effort go to code mitigations (more variation, per-proxy health/failover, `enableWarmup` default-on) vs. operator guidance (residential IPs, lower volume, separate machines)? This sets the bar for M1-03/M1-04 ambition and whether per-account UA variation is worth the engine-mismatch risk (currently an accepted limitation).
4. **Pending-approval product behavior.** Should admin-rejected pending posts be auto-re-queued, and should first-comment delivery move to permalink-primary so image-only/pending posts get comments? (Affects M1-04 scope and a known product defect.)
5. **Hidden-mode trade-offs.** Is the unavoidable taskbar icon acceptable, or is the second-virtual-desktop approach worth engineering? Is clipboard-paste-in-hidden-mode worth further work vs. accepting humanized typing?
6. **Distribution security bar.** Will the product be sold/distributed widely (→ code-signing M4-05, encrypted credentials M3-01, encrypted keys M3-05 become mandatory), or stay a single-operator tool (→ several hardening items can be documented-as-accepted)?

## 5. Suggested first sprint

1. **M1-01 — Image upload retry/abort.** Closes the most-cited blocker (silent image loss); directly protects campaign quality.
2. **M1-02 — Rotation-write halt.** One small change prevents the worst outcome for a posting tool — duplicate posts after a crash. Pairs with M2-01.
3. **M2-01 — Stop swallowing write/append errors.** Foundational: makes M1-02 and M2-02 surfaces visible.
4. **M1-05 — License enforcement decision + implementation.** Unblocks the entire licensing track.
5. **M1-03 — Composer/post-button/comment fallbacks + drift alert.** The single most likely cause of a silent "zero posts" overnight failure; builds the drift mechanism M1-04 reuses.
6. **M1-04 — Rate-limit/checkpoint/pending detection hardening.** Protects accounts from over-posting through undetected blocks.
7. **M2-05 — Clock-immune daily cap.** Small, high-value safety fix.
8. **M4-01 — Test runner + first suites.** Establishes the quality gate the rest leans on.

### Note on existing fallbacks (verifier correction)
The audit's adversarial verifiers flagged that the FB-automation code is **not** entirely without fallbacks — `openComposerByText`, `clickPostButton` (which scans all buttons), and `addFirstComment`'s permalink fallback already exist. The real M1-03/M1-04 gap is **fragility in the lower-level detectors** (`checkRateLimit`, `checkVerification`, `checkPendingApproval`, `dismissPopups`) and the **absence of automated drift detection** (`inspect-fb.js` is manual-only). Read the actual code before refactoring each detector.
