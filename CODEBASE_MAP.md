# za-post — Codebase Map

> Generated by a multi-agent read of the full codebase (10 reader agents + synthesis). Snapshot of the repo as of 2026-06-20.

## 1. What it is

**za-post** (packaged as "Za Post Comment Tool") is a Windows desktop application that automates posting content to **Facebook groups from many accounts at once**, optionally adding a first comment (typically a link) to each post. It is built on **Electron + Puppeteer (stealth)** and is a *restored, fully-editable reconstruction* of an originally-shipped app ("King") whose backend had been compiled to V8 bytecode.

**Who uses it:** a single operator (the app owner / a licensed customer) managing a stable of Facebook accounts — at scale, on the order of ~100 accounts, dozens of posts, and several groups per account — to push the same/varied marketing content into many groups on a schedule.

**End to end:** the operator logs each Facebook account in once (via a real visible browser window that captures cookies into a per-account Chromium profile), defines a library of **posts** (caption + optional images + optional first comment) and **groups** (Facebook group URLs/IDs), assigns groups and a posting mode to each account, tunes timing/anti-spam **settings**, and presses **Start**. The **orchestrator** then runs repeating cycles: each cycle it batches enabled accounts (running `parallelAccounts` concurrently), and for each account a **worker** launches a hidden Chromium profile, logs in (cookies/credentials), navigates to each assigned group, opens the composer, types/pastes the caption with human-like timing, uploads (optionally hash-perturbed) images, publishes, optionally adds a first comment, and records the outcome. The app detects rate limits, identity checkpoints, and pending-approval states, cools down or reassigns affected accounts, and writes a full audit trail. A **remote dashboard** (local Express server, optionally exposed over a Cloudflare tunnel) lets the operator monitor and control runs from a browser. An optional **VPS licensing system** can gate the app behind machine-bound license keys.

## 3. Architecture overview

The app is a single Electron process tree with three runtime planes — **UI**, **control/IPC**, and **automation** — plus two out-of-process services (Express remote server, VPS license server).

- **Electron main (`main.js`)** is the single source of truth and orchestrator-of-orchestrators: it owns IPC, the data store, browser automation for login/status, the licensing gate, and wiring of the Express server and Cloudflare tunnel.
- **Renderer (`renderer/`)** is the desktop UI; it talks to main **only** through the `electronAPI` bridge exposed by `preload.js` (context-isolated `ipcRenderer.invoke/on`).
- **Local server (`server.js` + `public/index.html`)** is a parallel control surface over HTTP, delegating all mutations back into main via injected hooks.
- **Automation (`automation/orchestrator.js` + `automation/worker.js`)** is the engine; the orchestrator schedules cycles/batches and the worker drives one Chromium per account against Facebook.
- **Shared libs (`lib/`)** provide persistence (`store.js`), licensing client (`license.js`), Chromium resolution (`chromium.js`), content variation (`spintax.js`), and image variation (`imageVary.js`).
- **VPS server (`vps-server/`)** validates and machine-binds license keys.

## 5. Core data & control flow

### Flow A — Campaign run: Start → cycles → post + first comment
1. **Operator presses Start** (renderer `startAutomation()`) after pre-flight (posts exist, groups exist, ≥1 account enabled+logged-in+with groups) → `electronAPI.startAutomation()` → `ipcMain.handle('start-automation')` (`main.js:868`) → `orchestrator.start(getData)` + `setRunActive(true)`.
2. **Orchestrator `_loop()`**: loads rotation state (`pcu-state.json`), then per cycle re-reads `data.json` via `getData`, checks connectivity (`isOnline()` racing google/facebook probes), and partitions enabled accounts into batches of `parallelAccounts` (staggered ~`idx*35s`).
3. **Per account**, `_postsForAccount()` selects posts by `postingOrder` (post-centric / random / sequence / *-unique) + `postFilter` + `postsPerGroup`; unique modes `claim` a post so parallel accounts don't collide. `_runAccount()` calls `worker.runAccount()` (≤2 retries).
4. **Worker** launches the account's hidden Chromium profile, authenticates (profile → cookies → credentials), and for each assigned group: dwells, `openComposer()`, uploads image (varied via `imageVary` when `varyImages`), enters caption (spintax-expanded when `varyContent`; clipboard paste → `humanType` fallback), `clickPostButton()` + `waitForPublish()`, then after `commentDelayMin..Max` seconds `addFirstComment()` (link uniquified by `varyLinks` when `randomizeLinks`).
5. **Outcomes** bubble up: worker → `_runAccount` → `_recordAccountOutcome` (daily count, rate-limit cool-down with exponential backoff) + `store.update` (account status/flags) + `store.appendReport` (audit row). Flags like `needs_login`/`checkpoint`/`rate_limited` emit `account-attention` → desktop toast.
6. **Cycle end**: persist `dealt` ids to `pcu-state.json`, optionally auto-delete published posts, check completion (all dealt in unique modes, unless `loopCampaign`) or 3-consecutive-zero-progress stall, then `_waitWithCountdown(waitInterval ±20%)`. `automation-progress`/`automation-summary` events drive the UI.

### Flow B — App launch & license check
1. `app.whenReady()` (`main.js:273`): `store.init`, clear interrupted logins, data-corruption check, kill orphan Chromium, set up logging, create orchestrator, register power events, start Express server (+ optional tunnel if `ENABLE_TUNNEL`/`settings.enableTunnel`).
2. **License gate (opt-in)** at `main.js:363`: `LICENSE_ON = ENABLE_LICENSE || settings.licenseEnabled`. If **off** → `createWindow()` directly (default). If **on** → `license.checkCached(userData, serverUrl)`: valid → main window; revoked → `showRevokedWindow()`; otherwise → `showLicenseWindow()`. Activation (`validate-license-async`) POSTs `{license, hwid}` to the VPS; on success caches to `userData/license.json`. Note: `get-license-info` (`main.js:955`) is a **separate hardcoded permissive stub** (`valid:true`, 9999 limits) used by the renderer for feature limits — it is *not* the gate.
3. **Auto-resume** (`main.js:381`): if the previous session left `run-state.json.active` set **and** real work remains **and** `resumeOnStartup===true`, the orchestrator restarts on `did-finish-load` (guarded to fire once).

### Flow C — Content/image variation pipeline (anti-spam)
Per `(account, group, post)`, the worker: expands caption/comment spintax via `spintax.expand()` (each group gets a distinct string when `varyContent`), perturbs each image via `imageVary.varyImage(src, "account|group|image")` (deterministic per group, shifting perceptual hash while looking identical, when `varyImages`), and appends a unique tracking param to comment links via `varyLinks()` (when `randomizeLinks`). Combined with per-account stable proxies, comment delay, daily caps, and exponential rate-limit cool-down, this reduces (but per the docs cannot eliminate) Facebook's spam-shape signal.

## 6. Data model & persistence

All persistent state lives under Electron **userData** (`%APPDATA%\za-post-restored\`, or `za-post-restored-{PROFILE}` when `--profile=`/`ZA_PROFILE` is set). `lib/store.js` is the only writer, using atomic temp→fsync→rename with `.bak` recovery and a serialized `update()` mutex.

**`data.json`** — `{ posts[], groups[], accounts[], settings{}, proxies[], useProxies }`:
- **post**: `{ id, caption, comment, imagePath(s)/imageUrl, commentImagePath/commentImageUrl }`.
- **group**: `{ groupId, name }` (id extracted from URL).
- **account**: `{ name, alias, enabled, status (not_logged_in|logged_in|rate_limited|checkpoint|error), lastMessage, assignedGroups[], postFilter (all|with-comments|without-comments), postingOrder (post-centric|random|sequence|post-centric-unique|random-unique), email/password (optional auto-login), proxy, fbUserId, fbName, lastChecked, daily{date,count}, rateLimitedUntil, rlStrikes, runCount }`.
- **settings** (`DEFAULT_SETTINGS`, `store.js:17`): pacing (`parallelAccounts:2`, `waitInterval:120`, `accountDelay:2`, `groupDelay:180`, `postsPerGroup:1`, `maxCycles:0`), behavior (`commentWithImage`, `autoDeletePosted`, `hideBrowser:true`, `loopCampaign`, `resumeOnStartup:false`, `launchOnStartup`, `enableTunnel:false`), anti-spam (`commentDelayMin:60`/`Max:180`, `dailyCap:0`, `enableWarmup`/`warmupRuns:5`, `varyContent:true`, `varyImages:true`, `randomizeLinks:true`, `staggerAccounts:true`, `rateLimitCooldownHours:4`).

**Other files:**
- `pcu-state.json` — `{ dealt:[post_ids], roundOffset }` rotation state (atomic + `.bak`).
- `run-state.json` — `{ active, ts }` crash/resume flag (synchronous fsync).
- `accounts/<name>/chrome-profile/` — per-account Chromium user-data dir (`sanitizeProfile(name, hidden)` clears session snapshots and pins off-screen bounds for hidden runs).
- `accounts/<name>/cookies.json` — imported/saved cookies (migrated cookies often stale → re-login needed).
- `storage/images/` — decoded post & comment images.
- `logs/automation.log` (rotated at 5MB → `.log.1/.log.2`) and `logs/run-report.jsonl` + `run-report.csv` (per-outcome audit, rotated at 5MB).
- `license.json` — cached `{ key, hwid, ts }`. `license-config.json` — overridden license server URL.

## 7. Licensing system

The licensing system is **opt-in** and spans the VPS server and the desktop client.

**Key generation (VPS):** `vps-server/gen-key.js` creates a 16-hex key formatted `XXXX-XXXX-XXXX-XXXX` and writes a record into `keys.json`: `{ hwid:null, revoked:false, expires:null, note, createdAt }`. `OWNER_KEY` env var seeds an owner record on first boot if missing.

**Validation (client → VPS):** `lib/license.validate(key, serverUrl)` POSTs `{ license, hwid }` to `<server>/api/validate` (default `http://144.91.127.7:3509`, 6s timeout). The server (`license-server.js:49`):
- unknown key → `{valid:false, message:'Invalid license key'}`;
- `revoked` → `{valid:false, revoked:true}`;
- expired (`expires` ms < `Date.now()`) → `{valid:false, message:'License expired'}`;
- **first activation** (no bound hwid) → binds `rec.hwid = hwid`, sets `activatedAt`, returns `{valid:true, 'Activated'}`;
- bound to a different machine → `{valid:false, 'already active on another device'}`;
- otherwise updates `lastSeen` → `{valid:true, 'OK'}`.

**Machine binding:** `hwid()` uses `node-machine-id.machineIdSync()` (fallback `os.hostname()`). Keys are first-activation-locked to one machine.

**Offline fallback:** if the server is unreachable, `offlineValid(key)` checks SHA256(uppercased key) against `OFFLINE_ALLOW` (a single hardcoded owner-key hash `2d12582d…`). Only the owner key works offline; customer keys require server reachability.

**Caching:** `activate()` writes `userData/license.json` `{key, hwid, ts}`; `checkCached()` re-validates the cached key on each boot, rejects on hwid mismatch, and clears the cache when the server reports `revoked`.

**Client gate & UI:** at boot (`main.js:363`) the gate runs only when `ENABLE_LICENSE`/`settings.licenseEnabled`. `validate-license-async` (renderer → main) drives activation; `license-validation-result` returns the verdict; success closes `license-window.html`, `revoked` shows `revoked.html` (WhatsApp contact + retry → `retry-license`). Server URL is configurable via `get-server-url`/`update-server-url` (`license-config.json`).

**Revocation:** `revoke.js` sets `revoked=true/false` (and optionally `--unbind` clears hwid) in `keys.json`; the next client `validate`/`checkCached` returns `revoked` and the app shows the revoked window. **Important:** the in-app `get-license-info` IPC (`main.js:955`) is an unrelated permissive stub returning lifetime/9999 limits and does **not** reflect real license state.

**API contract:** `POST /api/validate {license,hwid} → {valid, revoked?, message}`; `GET /api/keys?admin=<ADMIN_TOKEN> → full key store`.

## 9. Open questions for the user

What license model is desired (machine-bound seat, subscription, per-account cap)?

3. **Offline revocation:** since a slow server triggers offline fallback that can bypass revocation, is that acceptable, or should the client fail-closed on timeout for non-owner keys?
4. **Facebook selector drift:** are there known FB localizations/A-B variants affecting the `write something|what's on your mind` selectors, and should fallback selectors + automated selector-drift detection (diffing `inspect-fb.js` output) be added?

## 10. Where to start / suggested next steps

**Read first (in order):** `DOCS.md` (full technical reference + anti-spam audit) → `HANDOFF.md` (current status, 2026-06-20) → `OPERATIONS.md` (daily workflow). These are explicit ground truth.

**Code entry points:**
- **App boot & IPC surface:** `main.js` — `app.whenReady()` bootstrap (`273–397`) and the IPC handler block (`413–987`). Start here to understand control flow and the license gate (`363`).
- **Run engine:** `automation/orchestrator.js` `_loop()` (`~400–598`) and `_postsForAccount()` for posting-mode/rotation logic.
- **Facebook automation:** `automation/worker.js` `runAccount()` — the highest-risk, most-likely-to-break code; its DOM selectors are "the first thing to verify/tune on a live run."
- **Persistence contract:** `lib/store.js` `DEFAULT_SETTINGS` (`17`) and `normalize()`/`update()`/`save()` for the data shape and durability model.
- **Remote API:** `server.js` route table + the `hooks` it receives from `main.js`.

- Run `node scripts/test-fingerprint.js` (10 checks) and `node scripts/test-antispam.js` (27 checks) — both should pass; they cover fingerprint, content/image/link variation, daily-cap, and cool-down persistence.
- Run `node scripts/inspect-fb.js` for a logged-in account to capture current Facebook DOM (`scripts/fb-recon/`) and confirm composer/comment/post-button selectors still match `worker.js`. This is the canonical first step when posting breaks.
