# za-post — Core Enhancement Spec (posting · proxy · parallelism)

> Generated by a 28-agent analysis (13 enhancements verified). Build order in §4.

## 0. Summary

This spec hardens the three load-bearing subsystems of za-post: the per-group **posting pipeline** (`automation/worker.js`), **proxy handling/reliability**, and **parallel-account orchestration** (`automation/orchestrator.js`). The headline wins:

- **Stop losing whole accounts to one bad group.** Today a single transient nav/upload/CDP failure aborts an account mid-cycle. A per-group retry loop with classification (transient → retry, permanent → skip/abort) reclaims those groups.
- **Stop silent mis-publishes.** Escalate post-publish block-phrase detection to a real cooldown, detect FB's "select groups to share to" dialog before claiming success, and verify the first comment actually landed (input box emptied).
- **Make proxies observable and resilient.** Make the proxy-auth path auditable, give operators actionable parse-error guidance, add a net-new `lib/proxy.js` health module with optional cross-restart persistence, and expose a `/api/proxies/health` endpoint plus an offline `diagnose-proxies.js` CLI.
- **Reclaim idle parallel slots and equalize fairness.** Replace the fixed batch barrier with a dynamic concurrency pool (preserving all dealt/rotation/cap/cooldown gates), rotate start-order per cycle so account 0 isn't perpetually freshest, decay stagger over retries, and log pool utilization for tuning.

Invariants preserved everywhere: round-robin **dealt-id** persistence (`_persistDealt`), **rotation-state** halt-on-write-failure, per-account **daily cap** and **rate-limit cooldown** gates (all evaluated *before* an account is queued), offline-hold semantics, and the status-write mutex in `lib/store.js`.

## 1. Posting pipeline (`automation/worker.js`)

- **E-P1 — Per-group transient-failure retry with backoff** (M/High). Wrap caption→composer-open→publish (NOT navigation/upload — already self-retry) in a `groupRetry` helper, ≤2 retries, backoff 1.5s→3s. Classify errors: CDP/Timeout/ECONNRESET → retry; checkVerification/checkRateLimit → break account; 404/missing/"post button not found" → skip group. Before re-publish, query the feed to confirm the prior attempt didn't already land (duplicate guard). Soft cap ~2–3 min/group.
- **E-P2 — Detect & handle "share to multiple groups" dialog** (M/Med-High). BLOCKED on `scripts/inspect-fb.js` DOM capture (EN/FR/ES/DE) → add an `FB.groupSelect` array. Best-effort; a miss must never break the single-group flow.
- **E-P3 — Escalate post-publish block-phrase to cooldown** (S/Med). When the post-publish block scan trips, set `flag='rate_limited'`+`noRetry=true` and break the group loop, not just warn.
- **E-P4 — Verify first-comment delivery (box-emptied) + retry Enter** (M/Med). After Enter, confirm the exact box empties ≤6s; retry Enter once; emit `verified`/`not verified`/`failed`.

## 2. Proxy handling (`worker.js` + net-new `lib/proxy.js` + `server.js` + net-new `scripts/diagnose-proxies.js`)

New module `lib/proxy.js` (`ProxyHealthManager`): map `proxyStr → { lastFail, consecutiveFailCount, lastReason, cooldownUntil }`; `markFail/markOk/isOnCooldown/getStats/save/load` with >1h auto-prune. Never blocks a post.

- **E-X1 — Make proxy-auth path explicit & auditable** (S/Low). Track `authSetupOk`; log which auth method succeeded; warn if both fail. No reordering of the existing try/catch.
- **E-X2 — Actionable proxy parse-error guidance** (S/Low). `parseProxy` stays diagnostic-only (returns null); detect `ip:port` / `ip:port:user:pass` (no scheme) and append a suggested fix to the rejection log.
- **E-X3 — Persist proxy health across restarts** (S/Low). Instantiate the manager in orchestrator start; persist to `userData/proxy-health.json`; classify failures (407/refused/timeout/proxy-chain); prune >1h on load.
- **E-X4 — Proxy diagnostics: health endpoint + CLI** (S/Low). `scripts/diagnose-proxies.js` (offline) first; then `GET /api/proxies/health` behind the existing `X-Access-Token`.

## 3. Parallel-account orchestration (`automation/orchestrator.js`)

Hard requirement for every item: preserve `_persistDealt`, rotation halt-on-write-failure, daily-cap + cooldown gates (before queueing), offline-hold, and the `store.update` mutex.

- **E-N1 — Account isolation: formalize + regression-test** (S/Low, test-only). No code change; add a `parallelAccounts≥2` test proving A's crash/flag never affects B.
- **E-N2 — Pool-utilization metric** (S/Low). Return `durationMs` from `_runAccount`; log `[pool-stats] active/cpu/util`.
- **E-N3 — Fair stagger: rotate start-order per cycle** (S/Med). `_staggerRotation` rotates the active list each cycle; persist in `pcu-state.json`; reset on fresh run; header shows original order.
- **E-N4 — Smart stagger decay over retries** (M/Med). Per-account `retryCount`; stagger `= clamp(idx*35000 * 0.5^retryCount, 0, 35000)`; reset on success.
- **E-N5 — Dynamic concurrency pool (replace batch barrier)** (L/Med-High). `SimplePool` feeds all active accounts; launch next as a slot frees; stagger applied in dequeue; `_persistDealt` per completion; offline drains+holds; `_finish` drains in-flight. Pool is per-cycle.

## 4. Build order

1. E-N1 (isolation test) → 2. E-X1 + E-X2 → 3. E-P3 → 4. E-P4 → 5. E-N2 (`durationMs`) → 6. E-X3 (`lib/proxy.js`) → 7. E-X4 (endpoint+CLI) → 8. E-N3 (rotation) → 9. E-P1 (per-group retry, injection-tested) → 10. E-N4 (decay) → 11. E-N5 (dynamic pool, phased) → 12. E-P2 (blocked on DOM capture).

## 5. Key risks
- **E-P1** masking real blocks → strict classification (rate-limit/verification never retried) + duplicate-guard + soft cap.
- **E-N5** dealt/rotation/cap corruption → `_persistDealt` per completion; gates before queueing; crash/rotation tests must pass unchanged + a crash-mid-pool case.
- **E-P4** double-comments → box-empty check must target the exact element typed into.
- Re-run after each item: `retry`, `daily-cap`, `orchestrator-crash`, `orchestrator-rotation`, `fb-detection`, `security`, `store`, `settings` (all `npm test`).
